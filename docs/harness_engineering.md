# AIエージェントのシステム要件とハーネス・エンジニアリング

## 1. ハーネス・エンジニアリングとは何か？ (概念と哲学)

「ハーネス・エンジニアリング（Harness Engineering）」とは、AIエージェント（特にコーディングエージェント）を実用的なソフトウェア開発ライフサイクル（SDLC）に安全かつ効果的に組み込み、継続的に改善するためのシステム設計手法およびインフラストラクチャの構築を指します。

AIモデルを「脳」とするならば、ハーネス（Harness：馬具、安全帯、制御装置の意）は「体」と「外部環境」に相当します。
これまでのAI活用は「いかに良いプロンプトを書くか（プロンプト・エンジニアリング）」に主眼が置かれていましたが、エージェントが自律的に行動する時代においては、**モデルの周囲に「制約（Constraints）」「ツール（Tools）」「フィードバックループ（Feedback Loops）」を備えた構造化された環境を構築すること**が不可欠になります。

これにより、AIモデル自体の性能向上を待つのではなく、環境側で信頼性を担保し、タスクの成功率を高め、暴走を防ぐことが可能になります。

### 1.1 エンタープライズにおける重要性

企業がAIコーディングエージェントを本番運用に導入する際、「賢いモデルを選ぶこと」以上に「モデルをどう制御し、どう評価・改善するか」が重要になります。

*   **セキュリティと安全性の担保**: AIが本番環境や機密データに直接アクセスしたり、破壊的なコマンドを実行したりするリスクを防ぐため、隔離された実行環境（ガードレール）が不可欠です。
*   **信頼性と再現性の向上**: AIの出力は確率的であるため、そのままでは本番コードとして信頼できません。テストや静的解析といった自動化されたチェック機構を経由させることで、品質のばらつきを抑えます。
*   **継続的な改善（Continuous Improvement）**: エージェントがどこでつまずいたか（思考やツール呼び出しの履歴＝軌跡：Trajectory）をトラッキングし、システム全体を継続的にチューニングする仕組みが必要です。

## 2. エージェント導入を成功させる2つのアーキテクチャ・レイヤー

上記のような高度なハーネス・エンジニアリングを実現するためには、システムを単一のツールとして捉えるのではなく、以下の「2つのレイヤー」に分けて設計・評価する必要があります。

1. **第1部：エージェントのシステム・プリミティブ (Agent System Primitives)**
   エージェント自身が高度に自律稼働し、プロジェクトの文脈を理解するために持つべき「内蔵機能」や「拡張アーキテクチャ」です（いわば、より賢く動くための高度な馬具）。
2. **第2部：ハーネス・インフラストラクチャ (Harness Infrastructure)**
   エージェントを安全に走らせ、出力を検証するための「外部の実行環境や制御・評価基盤」です（いわば、安全に走らせるためのサーキットや監視システム）。

本ドキュメントでは、エンタープライズでAIエージェントを選定・運用する際に求められるこれら2つの要件について解説します。

---

## 3. 第1部：エージェントのコア機能とアーキテクチャ (Agent System Primitives)

単一のプロンプト入力にとどまらず、モダンなエージェント・システムが備えるべき拡張機能とエコシステムの構成要素です。

### 3.1. コンテキストと知識の標準化 (Context & Knowledge Management)
エージェントがプロジェクトの背景を深く理解し、適切な文脈で動作するための仕組みです。

*   **agents.md**: プロジェクトのルートディレクトリに配置し、そのプロジェクト固有のルール、アーキテクチャの方向性、コーディング規約、あるいはエージェントが振る舞うべき「ペルソナ」などを宣言的に定義するファイルです。これにより、エージェントが迷うことなくプロジェクトの「文脈」に沿ったコードを生成できるよう静的なコンテキストを注入します。
*   **MCP (Model Context Protocol)**: エージェントと外部のデータソースやツール（ローカルファイルシステム、社内Wiki、外部SaaS/APIなど）を標準化されたプロトコルで安全に接続し、動的にコンテキストを拡張する仕組みです。モデル自体を再学習させることなく、組織内の最新情報や機密データに安全にアクセスさせることが可能になります。

### 3.2. 機能拡張とライフサイクル制御 (Extensibility & Lifecycle)
エージェントの能力を柔軟に拡張し、その行動プロセスを安全に管理するための手法です。

*   **Agent Skills**: 「AWSインフラの設計」「特定の社内フレームワークの利用」「Google Workspaceの操作」など、特定領域に特化した専門的な知識（プロンプト）とツールセットをひとまとまりのモジュールとして定義し、タスクに応じてエージェントに「スキル」として着脱可能（Activate）にする概念です。
*   **Hooks**: エージェントのタスク開始前、シェルコマンド実行直前、ファイル変更時といったライフサイクルの特定の節目に介入（インターセプト）する仕組みです。ここに社内のセキュリティスキャンや承認フロー（ガードレール）をフックさせて破壊的変更を防ぐだけでなく、「コミット前の社内規約フォーマット」や「ビルド前処理の自動実行」といったワークフローの自動化にも活用されます。

### 3.3. マルチエージェント・オーケストレーション (Multi-Agent Orchestration)
複雑なタスクを単一のモデルで処理する限界を超え、複数エージェントの協調によってシステムをスケールさせるアーキテクチャです。

*   **Subagents (サブエージェント)**: メインのオーケストレーター・エージェントが、特定のタスク（例：「大規模コードの調査専門」「テストコードの作成専門」など）を特化型のサブエージェントに委譲（Delegate）する階層的アプローチです。これによりメインのコンテキストウィンドウの消費を抑えつつ、専門性の高い精緻な処理が可能になります。
*   **A2A (Agent-to-Agent Communication)**: 複数のエージェント同士が自律的に直接対話・協調する仕組みです。例えば、「実装エージェント」が書いたコードを「レビューエージェント」が検証・指摘し、人間を介さずに自律的な品質向上ループ（ピアレビュー）を回すといった高度なフローを実現します。

## 4. 第2部：ハーネス・インフラストラクチャ（実行環境と制御基盤）

エージェントの出力を信頼できるものにし、システム全体を継続的に改善（Continuous Improvement）するための外部インフラ設計です。

### 4.1. サンドボックス（安全な実行環境）
エージェントがコードを実行し、ツールを利用するための隔離された環境です。

*   **コンテナ化された使い捨て環境**: Dockerや一時的な仮想マシン（VM）を使用し、エージェントの作業ごとにクリーンな環境を提供します。
*   **権限の最小化（Least Privilege）**: エージェントにはタスク遂行に必要な最小限のネットワークアクセス権とファイルシステムへのアクセス権のみを付与します。
*   **仮想ファイルシステム（VFS）**: ファイルの変更を直接ホストに反映させるのではなく、VFS上でシミュレートし、人間またはCIが承認した後にマージするアプローチが取られます。

### 4.2. CI/CDパイプラインとの統合
AIエージェントは独立して動く魔法の箱ではなく、既存のCI/CDパイプラインの一部として統合されるべきです。

*   **プレコミット / プレPR 検証**: エージェントが生成したコードは、リポジトリにコミットされる前、あるいはPull Request（PR）が作成される前に、サンドボックス内で単体テスト、リンター、フォーマッターによる自動検証を受けます。
*   **ゲートキーパーとしてのCI**: テストに合格しない限りコードはマージされません。これにより、「コンパイルできないコード」がメインブランチに混入するのを防ぎます。
*   **イベント駆動型の実行**: Issueの作成、特定ラベルの付与、あるいは定期的なスケジュールをトリガーとしてエージェントを起動させます。

### 4.3. 評価フレームワーク（Evaluation Frameworks）
AIエージェントの性能を定量的かつ継続的に測定する仕組みです。

*   **標準ベンチマークの導入**: HumanEval（関数レベルのコーディング能力）やSWE-bench（実際のリポジトリでのIssue解決能力）などのベンチマークを利用して、ベースラインの性能を測ります。
*   **軌跡（Trajectory）の評価**: 「最終的な出力が正しいか」だけでなく、「目的のファイルを正しく検索できたか」「適切なツールを適切な順序で使えたか」といった、エージェントの思考・行動プロセス（軌跡）自体を評価します（例: LangSmithの活用）。
*   **LLM-as-a-Judge**: 人間がすべてをレビューするのはスケールしないため、別の強力なLLM（例: GPT-4o, Claude 3.5 Sonnet）を用いて、エージェントの出力が設計規約やセキュリティ要件を満たしているかを自動判定させます。

### 4.4. フィードバックループと自己修復（Self-Repair）
エラーが発生した際に、それを学習・修正の機会に変換するメカニズムです。

*   **エラーのフィードバック**: コンパイラエラー、Lintエラー、テストの失敗などの標準出力・標準エラー出力を直接エージェントに返し、「なぜ失敗したか」を分析させます。
*   **自己修復ループ**: エージェントはエラーメッセージを読み込み、自らコードを修正して再度テストを実行します（事前に設定されたリトライ上限まで繰り返す）。
*   **コンテキストの動的調整**: 失敗が続く場合、リトリーバル・システム（RAG）が提供するコンテキスト（関連ドキュメントやコードスニペット）を動的に変更・拡張し、エージェントが正解にたどり着きやすくする工夫（コンテキスト・エンジニアリング）を行います。

## 5. 結論

エンタープライズにおけるAIコーディングエージェントの真の価値は、単独のモデルの賢さだけで決まるものではありません。
**第1部**で挙げたような「エージェント自身がコンテキストを読み取り、自律的に能力を拡張・協調する機能」と、**第2部**で挙げたような「堅牢なサンドボックスとCI/CD統合によって出力を制御・評価するハーネス環境」が両輪として機能することで、初めて安全なスケールと継続的改善（Continuous Improvement）が実現します。

今後のツール選定においても、「これらのアーキテクチャ要件（機能・環境）をどこまでネイティブにサポートしているか、あるいは既存システムと統合可能か」が重要な評価基準となります。
